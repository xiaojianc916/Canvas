# ADR-005: Unified fatal incident architecture\n\n- Status: Accepted\n- Date: 2026-07-24\n- Owners: Desktop application composition root\n\n## Context\n\nHybrid Canvas previously implemented multiple unrelated failure paths:\n\n1. a static HTML startup card and bootstrap error renderer;\n2. an application-level React error boundary;\n3. a workspace-level UI error boundary;\n4. console-only global error and unhandled-rejection reporting.\n\nThese paths did not share state, diagnostics, recovery semantics or visual\npresentation. The workspace boundary also made a failure of the complete editor\nsurface appear to be a recoverable local feature failure.\n\nNative Rust process crashes form a separate physical boundary: once the native\nprocess terminates, the WebView cannot render a React error screen.\n\n## Decision\n\nHybrid Canvas uses one fatal incident model and one terminal fatal state.\n\n### Ownership\n\n- `fatal-incident.ts` owns normalization and the diagnostic snapshot.\n- `FatalIncidentController` owns terminal fatal state and deduplication.\n- `fatal-runtime.ts` owns the application singleton.\n- `fatal-collectors.ts` adapts browser and Vite failures.\n- `FatalErrorBoundary` adapts React render failures.\n- `FatalErrorHost` owns the only React global fatal presentation.\n- `pre-react-entry.ts` renders the same view model before React is available.\n- Rust diagnostics persist native panic reports for the next application launch.\n\n### Fatal state\n\nThe first distinct fatal incident becomes the primary incident.\n\nLater distinct incidents are counted and recorded but do not replace the primary\nincident. Repeated incidents with the same fingerprint are deduplicated.\n\nFatal state is terminal for the current renderer lifetime. It cannot be cleared\nby resetting component state. Recovery requires a reload or native restart.\n\n### Error classes\n\nThe global fatal surface is only for failures where the application cannot\nsafely continue:\n\n- bootstrap and runtime construction failure;\n- uncaught renderer exception;\n- unhandled Promise rejection;\n- root React render failure;\n- violated application invariant;\n- previous native process panic;\n- development-server compilation failure.\n\nExpected operational failures remain local:\n\n- document open, save and close failures;\n- settings failure;\n- native-window operation failure;\n- import or export validation failure;\n- optional feature and plugin failure;\n- image, media and font resource loading failure.\n\n### Diagnostics\n\nA fatal incident freezes:\n\n- incident ID and fingerprint;\n- error code, kind and lifecycle phase;\n- normalized technical message;\n- JavaScript and React component stacks when available;\n- source location;\n- bounded and redacted context;\n- recent bounded structured logs;\n- runtime and browser information.\n\nSensitive keys, credentials, bearer tokens and user-directory components are\nredacted before presentation.\n\nRust details remain local and cross the IPC boundary only through bounded,\ngenerated DTOs. Native and renderer reports use incident identifiers for\ncorrelation.\n\n### Presentation\n\nThe fatal screen is a full-window application state, not a card or dialog.\n\nIt uses:\n\n- one restrained warning icon;\n- concise user-facing language;\n- incident code and ID;\n- reload and copy-diagnostics actions;\n- collapsible technical details.\n\nNormal startup renders no loading card.\n\n## Rejected alternatives\n\n- keeping separate startup and runtime error pages;\n- resetting an Error Boundary to pretend that fatal state recovered;\n- treating the complete workspace as a recoverable feature boundary;\n- sending unrestricted Rust errors or filesystem paths to the renderer;\n- treating all resource-loading failures as application fatal;\n- allowing the fatal UI to depend on the normal workspace component tree;\n- maintaining multiple global error stores.\n\n## Consequences\n\n### Positive\n\n- startup, runtime, React, Vite and native recovery use one diagnostic model;\n- diagnostics are useful for debugging and bounded for safety;\n- global failure behavior is deterministic and testable;\n- local operational errors remain local;\n- no second canvas or document state model is introduced.\n\n### Costs\n\n- native crashes can only be presented on the next launch;\n- every new fatal source needs an explicit collector adapter;\n- recovery actions require lifecycle-specific testing;\n- diagnostic redaction rules must evolve with new data sources.\n\n## Verification\n\nThe decision is enforced by:\n\n- fatal incident unit tests;\n- fatal controller state-machine tests;\n- diagnostic buffer tests;\n- native crash recovery tests;\n- architecture checks preventing legacy boundaries and loading UI;\n- TypeScript strict type checking;\n- Rust tests and Clippy.\n
